> 简单的定时器事件管理 & SIGALRM


### 定时器事件队列 & SIGALRM

SIGALARM的作用是在设定的秒数之后, 内核会向进程发出SIGALARM信号, 进程可以绑定一个回调函数来执行一些到时动作. 其具有如下特点:

- 定时颗粒度为秒,因此精度不算高
- 默认行为是杀死进程
- 允许注册回调函数

由于其时间颗粒度是秒, 和系统滴答是对齐的, 也就是说, 到时时间可以表征为整数个系统滴答. 因此, 最简单的实现思路就是在TCB中新增计数字段, 在每次发生系统时钟滴答中断时, 递减TCB的计数字段, 达到0时, 执行回调函数. 但是,如果以后针对每种新机制的实现都采用这种缺什么补什么的方式,只会让TCB越来越臃肿和难以维护,后续添加新功能也会举步维艰.

实际上, 我们可以把SIGALRM信号归类到定时触发的机制下, 针对定时触发这个机制去实现一个子模块, 然后把所有定时触发相关的机制(包括SIGALRM)都纳入到这个子模块的管理下. 

Linux针对不同的精度要求, 把这类机制的管理分流到了两个不同的模块中管理:

| **特性** | **低精度定时器 (Timer Wheel)** | **高精度定时器 (hrtimer)** |
| --- | --- | --- |
| **核心算法** | **多级时间轮**，将不同时间范围的定时任务散列到不同层级，新增定时任务性能极高。 | **红黑树 (Red-Black Tree)**，按到期时间排序，确保最先到期的任务最快被处理。 |
| **精度** | **毫秒级 (ms)**，精度受系统时钟中断频率 `HZ` 限制。 | **纳秒级 (ns)**，依赖CPU的时间戳计数器等硬件提供高精度时钟源。 |
| **内核版本** | 从Linux内核早期开始使用。 | Linux 2.6.16 开始引入。 |
| **典型场景** | **网络/磁盘I/O超时**（大量设置但极少触发，如TCP重传定时器）。 | **实时多媒体、`nanosleep`、用户态POSIX定时器** (如 `timer_create`)。 |
| **内部结构** | 基于 `tvec_base` 结构，包含5个链表数组 (`tv1` ~ `tv5`)。 | 每个时钟ID对应一个 `hrtimer` 基础，管理其红黑树。 |

在我们的实现中, 由于处于开发早期阶段, 我们可以统一按照高精度定时器的设计框架来执行,后续引入更多使用场景后,扩展低精度定时器也不是什么难题. 我们把这么模块称为定时器事件管理模块.

每次在实现一个新的模块时, 我们都需要问自己这样几个问题:

- 模块对外提供什么功能?
- 模块的输入输出是什么?外部需要喂给模块什么?模块会吐回给外部什么?
- 模块对外需要暴露出的最少接口是多少个, 分别是什么? 什么是不需要暴露给外部的?

第一个问题很简单, 就是代替外部管理定时器事件. 外部告诉模块定时时间是多少, 到期时需要做什么, 然后把工作交给模块执行, 自己去执行其他任务.

第一个问题明确之后, 第二个问题也比较容易回答. 外部需要提供给模块的, 无非就是: 1. 定时时间是多少? 2. 需要执行什么动作? . 定时时间很直接, 用`usize`表征即可. 第二个需要怎么表征?最标准的做法,就是外部主动提供给模块一个回调函数, 模块在定时时间到期后,直接执行这个回调函数.这种做法的好处是:

- 模块内部的实现逻辑和外部的业务逻辑是解耦的, 因此模块的实现会变得异常简单
- 外部的业务逻辑对于模块内部而言是不可见的,也是不需要知道的, 后期维护更加容易, 模块可以单独存在,而不依赖外部模块.

基于以上考虑, 我们可以总结得到外部需要提供给模块的东西:

```rust
// 定时器队列元素
// 并不是所有定时任务都来自进程,因此不应该把Entry和进程绑定
pub struct TimerEntry {
    pub id: usize,     // 全局唯一的 ID
    pub deadline: usize,     // absolute deadline, 应该使用硬件滴答,即get_time(),切不可使用系统滴答
    pub trigger: Box<dyn TimerTrigger + Send + Sync>,
}

pub trait TimerTrigger: Send + Sync {
    fn execute(&mut self);  // 回调函数,由外部模块根据自己的业务场景实现
}
```

id用来唯一标识这个entry, deadline表示绝对到期时间, 其表征是用硬件滴答, 精度是纳秒, 对应我们前面提到的高精度定时器管理框架. trigger就是我们提到的回调函数, 任何线程安全的, 实现了`TimerTrigger`的对象都可以作为一个合法的trigger. 定时器到期后, 模块就会直接执行trigger的`execute`方法. 

在解决完菜谱问题后, 我们还需要解决容器问题, 即用什么数据结构来维护`TimerEntry`. 对于这类定时触发的场景, 我们可以总结这个容器需要具备的能力:

- 容器需要有优先级管理机制, 不同元素具有不同的优先级
- 在这个场景下, 越快到时的元素, 在容器中的优先级应该越高, pop时应该优先弹出

在定时器事件数量不多的情况下, 常用的算法是最小堆, 数量高时, 才会使用时间轮和红黑树算法. 因此我们可以发现, 定时器管理模块实际上可以用不同的容器来实现, 而选用的容器对于外部而言是不关心的, 是我们应当向外部屏蔽的信息. 因此, 我们把定时器管理模块所需要具备的能力抽象为一个trait(可以理解为一个基类), 由具体的容器来实现这个trait:

```rust
pub trait TimerContainer: Send + Sync {
    // pub fn new() -> Self;   // 这种不属于行为接口,trait只定义行为接口
                               // 并且其要求返回具体类型Self, 这在动态分发中(dyn)会造成编译错误
    fn add(&mut self, entry: TimerEntry);
    // Get the expired entry
    // fn tick(&mut self, now: usize) -> Option<Vec<TimerEntry>>;
    // tick会在时钟中断上下文调用,如果用Vec会造成堆分配,这在中断上下文中是比较危险的操作,因此改成如下签名
    // 调用方用循环不断取,直到取到None就不再调用,这样子比较合适
    // 实现者需确保：已过期但未返回的条目会在后续 tick(now) 中继续返回。
    fn tick(&mut self, now: usize) -> Option<TimerEntry>;
    // 去维护一个取消id的list,在tick中调用cancel检查这个事件是否被标记取消,被标记了就删除,而不是调用就当场删除
    fn cancel(&mut self, timer_id: usize);
}
```

这个trait同时回答了上面的第三个问题: 我们只需要提供上述接口, 同时, 对于背后的具体容器, 对外部屏蔽. 最后, 我们只需要针对某个具体容器去实现这个trait , 一个简单的定时器事件管理模块就算搭建完成了, 以最小堆为例:

```rust
impl TimerContainer for BinaryHeapBackend {
    fn add(&mut self, entry: TimerEntry) {
        self.heap.push(entry);
    }

    fn tick(&mut self, now: usize) -> Option<TimerEntry> {
        loop {
            if let Some(entry) = self.heap.peek() {
                if entry.deadline > now {
                    return None;     // 堆顶没到期,其余的肯定也没有
                }
            } else {
                return None;
            }

            let entry = self.heap.pop().unwrap();
            if self.canceled_ids.remove(&entry.id) {
                free_id(entry.id);
                continue;
            } else {
                return Some(entry);
            }
        }
    }

    fn cancel(&mut self, timer_id: usize) {
        // 不用在最底层去检查timer_id的合法性,这部分检查应当交给上层
        self.canceled_ids.insert(timer_id);
    }
}
```

有了上面的基础设施后, 我们来实现一个SignalTimerTrigger从而达到实现SIGALRM的效果:

```rust
pub struct SignalTimerTrigger {
    pub task: Weak<TaskControlBlock>,
    pub signal: SignalFlags,
}

// 只负责置位信号,不负责具体逻辑执行
impl TimerTrigger for SignalTimerTrigger {
    fn execute(&mut self) {
        if let Some(task) = self.task.upgrade() {
            task.inner_exclusive_access().signals |= self.signal;
        }
    }
}
```

信号场景是十分简单的, 就是在定时器到期后, 给对应信号位置位. 后续在增加更多的定时触发场景(比如定时sync块设备缓存)时, 我们只需要模仿上面的做法, 去实现一个具体的Trigger即可.
