> 实现简单的ls命令和cd命令

#### cd命令

在原始的rCore中, 只存在一级目录(即根目录), 所有程序都堆放在根目录下, 并且也没有引入路径更换和解析的功能, 这对于我们后续扩展其他功能是个挑战(如增加/tmp, /dev目录). 因此, 我们首先来完善这里的相关功能, 使得文件系统更加具备可用性.

首先要修改创建inode的create函数, 原本的create并不支持目录inode的创建, 我们要使得其支持目录创建, 同时引入Linux中的.和..特殊目录项:

```rust
pub fn create(&self, name: &str, type_: DiskInodeType) -> Option<Arc<EfsInode>> {
    ...
    get_block_cache(new_inode_block_id as usize, Arc::clone(&self.block_device))
        .lock()
        .modify(new_inode_block_offset, |new_inode: &mut DiskInode| {
            // new_inode.initialize(DiskInodeType::File);
            new_inode.initialize(type_);        // 根据指定类型初始化inode
        });
    /* 添加..和. */
    if type_ == DiskInodeType::Directory {
        get_block_cache(new_inode_block_id as usize, Arc::clone(&self.block_device))
            .lock()
            .modify(new_inode_block_offset, |new_inode: &mut DiskInode| {
                // increase size
                let block_needed = new_inode.blocks_num_needed(2 * DIRENT_SZ as u32); 
                let mut v = Vec::new();
                for _ in 0..block_needed { 
                    v.push(fs.alloc_data());   /* 分配一个块 */
                }
                new_inode.increase_size(2 * DIRENT_SZ as u32, v, &self.block_device);
                let dot = DirEntry::new(".", new_inode_id);
                new_inode.write_at(0, dot.as_bytes(), &self.block_device);
                let dotdot = DirEntry::new("..", self.inode_id);
                new_inode.write_at(DIRENT_SZ, dotdot.as_bytes(), &self.block_device);
            });
    }

    self.modify_disk_inode(|root_inode| {
        // append file in the dirent
        let file_count = (root_inode.size as usize) / DIRENT_SZ;
        let new_size = (file_count + 1) * DIRENT_SZ;
        // increase size
        self.increase_size(new_size as u32, root_inode, &mut fs);
        // write dirent
        let dirent = DirEntry::new(name, new_inode_id);
        root_inode.write_at(
            file_count * DIRENT_SZ,
            dirent.as_bytes(),
            &self.block_device,
        );
    });
    ....
}
```

self在这里我们就可以理解为父目录, 而new_inode就是我们即将插入的目录inode.

之后, 我们把测试程序从以前的打包进根目录改成打包进/test. INIT_PROC改成从这个目录去启动:

在启动阶段, 我们就不再需要调用list_apps, 这样子显示太多内容十分臃肿. 

打包进test后, 我们默认还是从ROOT_INODE进入shell的, 此时就需要添加cd命令的支持. cd命令本质上是user_shell程序做命令拦截, 在检测到合法的cd命令后, user_shell会去发起系统调用, 由系统调用完成工作目录的切换. 

切换工作目录十分简单, 就是修改TCB的cwd字段. 修改之后, 后续用户在shell上输入任何路径, 如果不以/, 也就是根目录开头, 那么底层都会从cwd开始解析. chdir的系统调用如下:

```rust
pub fn sys_chdir(path: *const u8) -> isize {
    let task = current_task().unwrap();
    let token = current_user_token();
    let path = translated_str(token, path);
    let cwd = {
        let inner = task.inner_exclusive_access();
        inner.cwd.clone()
    };
    let root = if path.starts_with('/') { &ROOT_INODE } else { &cwd };
    match resolve_path(root, &path) {
        Ok(new_dir) if new_dir.is_dir() => {
            task.inner_exclusive_access().cwd = new_dir;
            0
        }
        _ => -1
    }
}

/* ================================================================== */
// shell
if cmd == "cd" {
    // ...
    if chdir(new_path) == -1 {  // 直接修改 shell 进程的 cwd
        // error
    }
}
```

也就是说, 输入cd后, shell进程的cwd会被修改. 然后, 在输入程序名字后, 就会触发fork和exec系统调用. fork中, 子进程, 也就是被执行的程序, 会继承shell进程的cwd字段. 之后, exec中就会根据修改的cwd正确找到程序的位置:

```rust
pub fn sys_exec(path: *const u8, mut args: *const usize) -> isize {
    // ...
    let task = current_task().unwrap();
    let inner = task.inner_exclusive_access();
    let cwd = inner.cwd.clone();   // ← 拿到子进程继承来的 cwd
    drop(inner);
    // ...
    if let Some(app_inode) = open_file(path, OpenFlags::RDONLY, &cwd) {
        // 找到了，加载并执行
    }
}
```

也就是说, 实现到这里, 我们已经支持把可执行程序放置在任意路径下, 并且用户可以从任意路径cd到这个路径下执行程序.

#### ls命令

有了前面的基础之后, 实现ls命令也是水到渠成的. ls命令的本质是打印当前工作目录的情况. 因此, 其背后就是sys_getdents系统调用:

```rust
/// sys_getdents: list directory contents (kernel-side print)
pub fn sys_getdents(path: *const u8, _buf: *mut u8, _len: usize) -> isize {
    let task = current_task().unwrap();
    let token = current_user_token();
    let path = translated_str(token, path);
    let inner = task.inner_exclusive_access();
    let cwd = inner.cwd.clone();
    drop(inner);
    match list_dir(path.as_str(), &cwd) {
        Some(names) => {
            for name in &names {
                print!("{} ", name);
            }
            println!("");
            names.len() as isize
        }
        None => -1
    }
}

pub fn list_dir(path: &str, cwd: &Arc<OSInode>) -> Option<Vec<String>> {
    // 绝对路径从根目录开始，相对路径从 cwd 开始
    let root = if path.starts_with('/') { &ROOT_INODE } else { cwd };
    // 解析路径，找到目录对应的 inode
    let dir = resolve_path(root, path).ok()?;
    let mut names = Vec::new();
    let mut offset = 0;
    // 逐条 readdir，直到返回 None
    while let Ok(Some(entry)) = dir.readdir(offset) {
        names.push(entry.name);
        offset += 1;
    }
    Some(names)
}
```

标准的做法其实是把信息传到用户态, 由用户态来决定怎么处理这些东西. 现在我们为了简化, 就直接在内核态打印这些信息了. 后续会对这个函数做重构.

现在的cd和ls支持在任意目录下执行, 类似Linux中PATH的效果. 但是实际上这种机制的实现目前是依赖在user_shell中去拦截而实现的, 这其实是十分不靠谱和欺骗自己的行为. 后续, 我们会考虑引入PATH等机制, 使得用户对文件系统的体验更加丝滑.
