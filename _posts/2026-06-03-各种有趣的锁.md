> 各种有趣的锁


在某篇文章中, 我们讲了页缓存的实现, 但是没有展开讲上面用到的各种锁. 这篇文章中, 我们来讲解一下这些锁. 

进入正式的讲解之前, 我们先对Rust提供的sync原语做一些辨析.

构成Rust所有权系统的最核心机制就是编译器的借用检查器, 即&和&mut, 以及生命周期标注. 主要会检查以下三点:

- 引用的生命周期是否有超过被引用数据的可能, 如果有, 需要手动标注生命周期
- 任意时刻可以存在任意多个不可变引用或者一个可变引用, 即&和&mut无法同时存在

比如对于引用问题, 下列代码就无法通过编译期的检查:

```rust
let mut a = 10;
let r1 = &a;
let r2 = &mut a;  // 错误！不能同时存在不可变和可变引用
println!("{}", r1);
```

对于变量的使用, 一般是涉及到move还是复制的问题. Rust中有两种trait是专门来解决这个问题的, 即Copy和Clone.

实现了Copy trait的变量, 就意味着其可以安全地通过memcpy来创建副本, 而不用转移资源的所有权, 这通常对于那些可以安全bit-wise copy的类型是成立的, 但是对于那些在堆上持有资源, 比如一个指针而言, 则是不成立的.

Clone则是我们传统意义上理解的深拷贝, 它不仅会复制表面数据, 连背后的资源也会复制一份.

实现了Copy trait的变量在赋值时, 优先触发Copy而不是move, 即此时原本的变量不会失效. Clone则是要依赖手动调用.clone()方法.

对于自定义的结构体, 即使成员都实现了Copy trait, 编译器也不会帮我们自动推导, 而依赖我们手动声明.

```rust
#[derive(Copy, Clone, Debug)]
struct Vec2 { x: f64, y: f64 }

fn consume(v: Vec2) { println!("{:?}", v); }

let a = Vec2 { x: 1.0, y: 2.0 };
let b = a;          // a 被复制，a 依然可用
consume(a);         // 传参时 a 被复制，a 仍然可用
println!("{:?}", a); // 完全合法
```

上面都是编译期的静态检查相关的机制. 对于引用这种情况, 静态检查是可以覆盖到任何关于引用的错误使用问题的(不考虑unsafe等需要程序员自己承诺保证安全性的情况). 但是, 正是由于其全覆盖的问题, 也会导致其检查策略极其严格和保守, 从而失去了一定的灵活性. 

比如说, 考虑一个简单的读写锁设计问题: 要求读访问并发和写访问互斥. 如果只用静态引用去尝试做并发控制, 则不用任何东西包裹T:

```rust
struct RwLock {
		data: T,
		inner_lock: spin::Mutex<LockState>
}

impl RwLock {
		pub fn read(&self);
		pub fn write(&mut self);
}
```

由于是并发访问, 外部肯定同时拿到的是&RwLock, 而不是&mut RwLock, 这就直接导致了write这个签名是不成立的(&无法传入&mut这个形参)

为此, 我们需要修改write的签名为&self, 但是对于静态检查, &RwLock是无法得到&mut data和&mut inner_lock的, 即不可变引用无法得到其内部成员的可变引用, 即内部不可见性. 因此, 如果在纯静态检查的情况下去实现读写锁是举步维艰的.

从上面的分析可以看到, 最核心的问题就是怎么从&RwLock拿到内部的可变引用. 这就是运行时检查器登场的时候了.

所谓的运行时检查, 在Rust中一般称作内部可变性模式, 即可以通过不可变引用获取到内部字段的可变引用, 从而对其进行修改. 这个模式中最核心的就是`UnsafeCell<T>` .

`UnsafeCell<T>`是唯一允许通过不可变引用获取内部可变引用的类型, 其本身不会做任何运行时检查. 但是, `UnsafeCell<T>`并不是直接返回引用, 而是返回可变裸指针, 引用这一步需要我们手动进行转换. 说不如做, 来看一段代码: 

```rust
use std::cell::UnsafeCell;

fn main() {
    let cell = UnsafeCell::new(5);

    // 获取两个可变裸指针，它们可以共存
    let p1: *mut i32 = cell.get();
    let p2: *mut i32 = cell.get();

    // 通过裸指针并发修改（借用检查器完全不管）
    unsafe {
        *p1 = 10;
        *p2 = 20; // 两个裸指针同时存在，编译顺利通过
    }

    // 现在尝试把裸指针转换为 &mut 引用，再同时使用
    // 这会被借用检查器阻止
    unsafe {
        let r1: &mut i32 = &mut *p1; // 创建第一个可变引用
        let r2: &mut i32 = &mut *p2; // 编译错误！不能同时存在两个可变引用
        let r3: &i32 = &*p2;         // &和&mut同时存在, 不允许
        *r1 = 1;
        *r2 = 2;
    }
}
```

也就是说, `UnsafeCell<T>`通过`get`方法获取到的是对象的”可变裸指针”, 这里的可变不是说裸指针是可变属性的, 而是说它具备修改内部对象的能力. 裸指针就意味着其不纳入Rust的借用检查框架下, 而更像是C语言中的一个普通指针, 此时, 我们可以在unsafe块里对他们进行并发修改, Rust不会阻止我们. 

通常为了安全性, 我们会把可变裸指针转化成引用, 从而纳入到Rust的借用检查框架下. 通常的做法就是解引用后再引用, 即`&*`和`&mut*`, 会分别返回`&self`和`&mut self`. 可以看到, 裸指针中的mut并不一定会继承到引用中, 这完全取决于我们需要的是不可变引用还是可变的. 可以说, Rust提供了灵活性和安全性的选择, 裸指针的使用可以十分灵活, 但是需要程序员自身保证程序的安全, 如果不需要这种灵活性, 那么纳入借用检查的体系下去编写代码是更好的选择.

`UnsafeCell<T>`作为最灵活的内部可见性类型, 是其他内部可见性类型的基础. `RefCell<T>`基于`UnsafeCell<T>`扩展, 提供了运行时的借用检查机制. 看下面一段代码:

```rust
use std::cell::RefCell;

fn main() {
    let cell = RefCell::new(42);

    let r1 = cell.borrow();           // 获取共享借用，运行时计数增加
    let r2 = cell.borrow_mut();       // 同时请求可变借用，运行时发现计数不为 0 → panic!
    
    println!("{}", r1);
    // 即使不打印 r1，上面的 r1 还活着，borrow_mut 就会 panic
}
```

将变量用`RefCell<T>`包裹后, 引用由`RefCell`返回的guard进行仲裁, 编译器不再介入. 因此, 上面一段代码是可以通过编译检查的, 只不过会在运行的时候panic.

`RefCell`实际上仅在单线程环境下是安全的, 如果在多线程场景下, 可能出现同时获取`borrow_mut`, 或者borrow+borrow_mut的情况, 此时就会panic.  如果想在单核环境下去用这个做同步, 就需要内核态时关闭中断, 且没有抢占, 否则就有可能在嵌套的中断上下文下或者抢占下重复获取可变引用导致错误. 不过既然都没有中断和抢占了, 那这个同步机制其实就有点掩耳盗铃了, 顶多起到了帮助程序员检查代码逻辑错误的工具.

如果用`UnsafeCell<T>`, 就有可能获取到两个&mut T, 并且不会报错:

```rust
pub struct UPSafeCell<T> {
    inner: UnsafeCell<T>,
}

impl<T> UPSafeCell<T> {
    pub fn get_mut(&self) -> &mut T {
        unsafe { &mut *self.inner.get() }
    }
}

let cell = UPSafeCell::new(42);
let r1 = cell.get_mut();  // 创建第一个可变引用
let r2 = cell.get_mut();  // 再次创建第二个可变引用
*r1 = 1;
*r2 = 2; // 两个 &mut 指向同一内存，未定义行为！
```

而使用`RefCell<T>`, 就可以检测出上面的未定义行为, 通过报错显式告诉程序员这里存在嵌套调用. 总而言之, `RefCell<T>`用于单线程环境下的运行时借用检查.

理解了RefCell<T>, 我们其实就可以理解RwLock的设计思路. RefCell<T>本质上是用UnsafeCell<T>提供内部可见性(由&UnsafeCell<T>获取到&mut T或者&T) + 严格的仲裁机制(试图并发获取&mut T时直接panic). RwLock仍然通过UnsafeCell<T>提供内部可见性, 但是使用比较宽松的仲裁机制(试图并发获取&mut T时, 阻塞未首先抢占到的线程, 等待信号量释放后再获取).  

因此, 我们给出如下的RwLock实现:

```rust
pub struct RwLock<T> {
    data: UnsafeCell<T>,                    /* 读锁可以共存, 写锁只能有一个 */
    inner_lock: UPSafeCell<LockState>,      /* 一般访问这个就是要去修改的, 所以直接用RefMut保护 */
    read_waiters: WaitQueue,
    write_waiters: WaitQueue
}

// send表示该类型的值可以安全地转移所有权到另一个线程
// sync表示该类型的引用&T可以安全地被多个线程共享
// 这两个声明都是为了能够通过编译器检查而已,因为我们并不存在真正的多线程环境
// 其实就是把安全职责从编译器转到实现者手中, 让编译器放行安全检查
unsafe impl<T: Send> Send for RwLock<T> {}
unsafe impl<T: Send + Sync> Sync for RwLock<T> {}

struct LockState {
    active_writers: usize,
    active_readers: usize
}
```

data的UnsafeCell<T>提供内部可见性, inner_lock用UPsafeCell在单核环境下可以检测代码逻辑错误(由于关中断+无抢占, 实际上并不存在真正的并发获取锁, 如果允许中断, 其内部的RefCell会panic掉嵌套获取锁的情况). 

目前的这个实现和多核下的实现已经十分接近, 只不过多核下需要把inner_lock信号量用阻塞式或者自旋式的Mutex保护, 或者干脆使用原子变量.

如果考虑用自旋锁保护, 就需要在获取锁之前关闭本地中断. 因为如果没有关闭中断, 而线程在锁的上下文中由于本地中断被切到了中断上下文, 当中断上下文很不幸地也尝试获取同一把锁, 就会导致CPU陷入死锁而进入永远的忙等待. 这一现象的根本原因是因为中断上下文是不受调度器管理的, 没有人可以把中断上下文切回获取锁的线程, 从而让其可以释放手中的锁. 因此, 主流的做法都是在获取自旋锁时关闭本地中断, 如Linux:

```rust
void update_data_process_context(void)
{
    unsigned long flags;

    /* 关闭本地中断，同时获取自旋锁 */
    spin_lock_irqsave(&my_lock, flags);

    /* 临界区：安全地读写 my_data */
    my_data.counter++;
    my_data.value = some_computation();

    /* 释放锁并恢复到之前的中断状态 */
    spin_unlock_irqrestore(&my_lock, flags);
}
```

定义好RwLock之后, 通常获取锁的方式是遵循RAII的原则, 即获取锁的时候, 会返回一个Guard, 通过Guard来访问内部数据, 同时在释放Guard时释放锁. 因此, 我们可以定义如下的两个Guard:

```rust
pub struct RwLockReadGuard<'a, T> {
    data: &'a T,    /* points to a read-only reference of data */
    lock_state: &'a RwLock<T>,
}

pub struct RwLockWriteGuard<'a, T> {
    data: &'a mut T,
    lock_state: &'a RwLock<T>,
}
```

Guard内部持有RwLock的引用, 以及数据的可变或者不可变引用. 持有RwLock是为了在Drop trait中去修改信号量, 即锁的释放, data的引用则是外界访问被锁保护的数据的方式. 我们在Guard上实现几个trait:

```rust
// let gg: RwLockReadGuard;
// gg.data.some_field -> (simplified to) gg.some_field
impl<T> Deref for RwLockReadGuard<'_, T> {
    type Target = T;
    fn deref(&self) -> &Self::Target {
        self.data
    }
}

impl<T> Deref for RwLockWriteGuard<'_, T> {
    type Target = T;
    fn deref(&self) -> &Self::Target {
        self.data
    }
}

impl<T> DerefMut for RwLockWriteGuard<'_, T> {
    fn deref_mut(&mut self) -> &mut Self::Target {
        self.data
    }
}

// Drop trait, automatically drop the lock reference
impl<T> Drop for RwLockReadGuard<'_, T> {
    fn drop(&mut self) {
        let mut state = self.lock_state.inner_lock.exclusive_access();
        state.active_readers -= 1;
        if state.active_readers == 0 {
            drop(state);
            self.lock_state.write_waiters.wake_one();   /* 读者为0了, 可以唤醒写的 */
        }
    }
}

impl<T> Drop for RwLockWriteGuard<'_, T> {
    fn drop(&mut self) {
        let mut state = self.lock_state.inner_lock.exclusive_access();
        state.active_writers = 0;
        drop(state);
        self.lock_state.read_waiters.wake_all();
    }
}
```

Deref和DerefMut只是为了方便数据的访问, 重点是两个Guard的Drop trait. 其实就是修改信号量, 并且在边界条件下(没有读请求了, 或者没有写请求了)唤醒阻塞的进程.

之后, 我们就可以来实现RwLock了:

```rust
impl<T> RwLock<T> {
    pub fn new(data: T) -> Self {
    }

    pub fn read(&self) -> RwLockReadGuard<'_, T> {
        loop {
            let mut state = self.inner_lock.exclusive_access();
            // 只要没有写的, 就可以顺利获取到锁
            if state.active_writers == 0 {
                state.active_readers += 1;
                drop(state);
                return RwLockReadGuard {
                    data: unsafe { &*self.data.get() },
                    lock_state: self,
                }
            }
            // 及时归还锁
            drop(state); 
            self.read_waiters.wait();   /* 这个调用会导致当前进程被切走, 醒来后重新检查, 如果没有人拿写锁就可以成功获取到读锁了 */
        } 
    }

    // &self + UnsafeCell达到同时调用 + 没拿到锁的阻塞的效果
    pub fn write(&self) -> RwLockWriteGuard<'_, T> {
        loop {
            let mut state = self.inner_lock.exclusive_access();
            // 没有写的, 并且也没有读的, 就可以获取写锁
            if state.active_writers == 0 && state.active_readers == 0 {
                state.active_writers = 1;
                drop(state);
                return RwLockWriteGuard {
                    data: unsafe { &mut *self.data.get() },
                    lock_state: self,
                };
            } 
            // 如果已经存在持有写锁的, 或者存在读锁, 就算失败
            drop(state);
            self.write_waiters.wait();  // 被唤醒后重试
        }
    }
} 
```

还是需要强调, 我们目前并不存在真正的并发访问, 但是这个实现在迁移到多核的时候, 不需要改动很多. 但是我们目前采用的是阻塞式的, 而标准库一般采用的是自旋式, 后续引入SMP后, 我们可以尝试从阻塞迁移到自旋.
