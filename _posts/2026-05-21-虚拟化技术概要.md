> QEMU KVM QEMU/KVM Hypervisor

物理机上安装一个操作系统，在操作系统上安装一个虚拟机管理软件，利用虚拟机管理软件启动多个客户机实例。客户机在发起系统调用后，会被虚拟机管理软件接收，再交由宿主机操作系统处理后返回，从而实现一个宿主机运行多个客户机实例。上述的虚拟机管理软件一般被称为VMM(Virtal Machine Manager)。目前主流的虚拟化方式主要是半虚拟化和全虚拟化，其中全虚拟化又分为二进制翻译和硬件辅助的虚拟化。

- 基于二进制翻译的全虚拟化
    
    这种虚拟化把VMM直接安排在ring0。客户机操作系统运行在ring1，用户态程序运行在ring3。在发生系统调用时，用户态程序先陷入客户机操作系统，客户机操作系统再陷入VMM，由VMM把敏感的指令替换为等价的安全指令组，以及架构的翻译(比如x86上跑ARM)，因此上下文切换开销很大，效率比较低。并且，VMM需要定期扫描当前的指令是否是敏感的，并予以替换，因为没有别的东西来代替它审查。而99%的指令都是不需要审查的，因此这种方式极大地降低了整个流水线的速度。
    
    ![image.png](/img/in-post/2025-05-21-虚拟化技术概要/image-2.png)
    
- 基于硬件支持的全虚拟化
    
    这种方式需要运行在提供硬件虚拟化支持的CPU上。这种CPU分为非root和root两种工作方式，每种方式都有4个独立的ring0-ring4特权级。开启虚拟化后，CPU工作在非root模式下，在遇到需要VMM处理的指令时，会切换为root模式，把控制权交给VMM，由VMM执行完对应的指令后再切换为非root模式。这种方式直接消灭了二进制翻译下反复审查的开销以及部分切换开销，变成CPU主动通知VMM的方式，效率提升许多。
    
    ![image.png](/img/in-post/2025-05-21-虚拟化技术概要/image-3.png)
    
- 半虚拟化
    
    半虚拟化通过修改操作系统内核，替换无法虚拟化的指令。仅支持linux，并且难度较大。
    
    ![image.png](/img/in-post/2025-05-21-虚拟化技术概要/image-4.png)
    

### QEMU & KVM[(1)](https://www.cnblogs.com/wujuntian/p/16294898.html)[(2)](https://www.cnblogs.com/edver/p/14662609.html)

QEMU是一个纯软件实现的虚拟化系统，它包办了虚拟化过程全链路：I/O, 内存，CPU, 设备。QEMU作为宿主机的一个VMM，把客户机的指令翻译为宿主机的对应指令后执行，再返回给客户机。这种方式的性能较差，不过可以运行和宿主机不同架构的SoC。

KVM则是基于Linux内核的开源虚拟化解决方案，其从Linux2.6开始被引入了内核，相当于内核引入的一个虚拟机管理模块。KVM仅提供针对CPU和内存的虚拟化，而不负责I/O和设备的虚拟化。

QEMU / KVM则是融合了两者的职责，由KVM提供内存和CPU的虚拟化，而QEMU则负责设备和I/O的虚拟化。

- QEMU-KVM属于硬件辅助的全虚拟化技术，CPU具有root和non-root两种操作模式。VMM（KVM）和宿主机内核运行于root模式下的Ring0，QEMU-KVM运行于root模式下的Ring3；虚拟机内核运行于non-root模式下的Ring0，虚拟机中的用户态程序运行于non-root模式下的Ring3。
- **一个KVM虚拟机就是一个QEMU-KVM进程，与常规Linux进程无异，与其他Linux进程一样被Linux进程调度器调度。**KVM虚拟机的CPU作为线程运行在QEMU-KVM进程的上下文中，KVM虚拟机的内存则是QEMU-KVM进程地址空间的一部分。
- KVM本身不执行任何硬件模拟，IO设备等硬件由QEMU-KVM进行模拟。
- KVM向上层提供/dev/kvm接口，/dev/kvm是一个标准的字符设备，通过ioctl接口控制；QEMU-KVM通过调用/dev/kvm设备的ioctl接口，对虚拟机进行相关控制，比如创建虚拟机、创建vCPU、运行虚拟机等。（**QEMU-KVM通过ioctl和/dev/kvm与KVM内核模块交互**）
- 换句话说，在不涉及I/O访问时，虚拟机的指令是直接跑在物理CPU上的(non-root模式提供的虚拟化机制)，和宿主机共享相同的原生性能。只有在涉及I/O访问时，才会触发VM-Exit，把控制权交给宿主机的KVM, KVM再把它交给用户态的QEMU-KVM(负责设备模拟)。

除了KVM, QEMU也支持其他类型的hypervisor，下表是官方文档给出的QEMU所支持的一些hypervisor: 

| **Accelerator** | **Host OS** | **Host Architectures** |
| --- | --- | --- |
| KVM | Linux | Arm, MIPS, PPC, RISC-V, s390x, x86 |
| Xen | Linux (as dom0) | Arm, x86 |
| MSHV | Linux (as dom0) | x86 |
| Hypervisor Framework (hvf) | MacOS | x86, Arm |
| Windows Hypervisor Platform (whpx) | Windows | Arm, x86 |
| NetBSD Virtual Machine Monitor (nvmm) | NetBSD | x86 |
| Tiny Code Generator (tcg) | Linux, other POSIX, Windows, MacOS | Arm, x86, Loongarch64, MIPS, PPC, s390x, Sparc64 |

前面都是一些不同的hypervisor，最后一个TCG则是QEMU的兜底机制。在用户主动关闭hypervisor，或者物理机的CPU不支持虚拟化时，QEMU就会使用TCG，即前面所提到的动态二进制翻译技术。在这种工作方式下，QEMU既模拟外设，也模拟CPU和内存，因此性能比较差，但是优点是可以跨架构运行程序。 

在现代服务器、云服务等场景，虚拟化的目的是为了榨干硬件性能，而不是为了跨架构运行程序，因此主流的方案都是选择基于硬件的全虚拟化，在这种方案下，只能运行同架构程序，但是客户机性能和原生性能相差无几。在下文的诸多讨论中，我们一般都会假设这种方案，而不去讨论TCG。

在虚拟化环境下，客户机使用其自身的虚拟地址，即GVA。而客户机的操作系统则负责把GVA翻译为（页表可以在硬件，或者由软件模拟）GPA。但是，GPA并没有办法用来真正地访存，因为说到底GPA也只是host os中，guest os所属的进程的一个虚拟地址，这个虚拟地址需要进一步经过宿主机的MMU翻译后才是真正意义上的物理地址。我们把这几个地址总结如下：

- GVA(Guest Virtual Address)：客户机虚拟地址
- GPA(Guest Physical Address)：客户机物理地址，即宿主机的某个进程的虚拟地址
- HPA(Host Physical Address)：宿主机物理地址，真正的物理地址

在这一整条翻译链中，目前有两种方案的存在，一种是所谓的**影子页表**，其完成的是一步到位式的翻译，即直接从GVA → HPA, 这个过程由VMM维护，其会维护每个客户机进程的影子页表。一般而言，这个转换过程并不像正常的CPU般有一个硬件MMU完成工作，而是纯软件模拟查表，因此十分占用CPU时间，效率较低。另一种则是intel提出的EPT翻译方式，其类似MMU,由硬件来完成整个翻译链条，因此效率更高。
