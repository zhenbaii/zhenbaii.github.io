> 初步引入VFS中间层抽象

初步引入VFS

Linux中的VFS是一个纯软件的中间层抽象, 其核心作用是向上层提供统一的文件系统抽象, 屏蔽底层的操作.**无论底层是 ext4、XFS、NFS、proc 还是 tmpfs，用户程序看到的都是同一套“文件”模型，可以用 open / read / write 等统一接口操作**。

VFS使用内存中的一些数据结构以及操作函数表来描述文件系统以及其中的文件, VFS的下游, 也就是具体文件系统, 需要去实现这些数据结构提供的接口来接入.

| **对象** | **关键结构体** | **含义** |
| --- | --- | --- |
| 超级块 | `struct super_block` | 代表一个已挂载的文件系统实例，包含该文件系统的元信息（块大小、设备、根 dentry 等）。 |
| 索引节点 | `struct inode` | 代表一个具体的文件、目录或设备，存储权限、大小、时间戳等元数据，以及指向实际数据块的地址信息。 |
| 目录项 | `struct dentry` | 代表路径中的一个分量（目录或文件的名字），将文件名映射到对应的 inode，是路径解析和缓存的核心。 |
| 文件对象 | `struct file` | 代表进程打开的一个文件，记录当前读写位置（`f_pos`）、打开标志（O_RDONLY 等）以及对应的 dentry 和 inode。 |

总的来说, 在初期引入VFS类似的机制, 可以让我们后面在引入大量其他模块时收获前所未有的便利性. 在引入新模块时, 我们只需要专注开发模块内的逻辑实现, 以及实现VFS提供的接口, 就可以无痛对接上层. 如果没有引入这种中间层抽象, 上层就需要记住每个底层模块所提供的独特的API, 这对于开发流程来说是灾难性的.

为了引入VFS, 首先要做的就是重构原本的一些底层数据结构. 首先是Inode. 原本Inode结构体用来表示easy-fs这个文件系统自身的Inode, 但是这显然是不合理的. Inode作为一个通用的, 管理文件系统的抽象, 不能被某一个具体的文件系统所独占. 因此, 首先要做的第一步就是把Inode还给VFS, 让easy-fs定义自己的Inode:

```rust
// 运行时抽象,驻留在内存中, DiskInode则是驻留在磁盘中, 可以理解为一个Inode背后对应一个在磁盘上的DiskInode
pub struct EfsInode {
    block_id: usize,
    block_offset: usize,
    inode_id: u32,
    fs: Arc<Mutex<EasyFileSystem>>,
    block_device: Arc<dyn BlockDevice>,
}
```

这个`EfsInode`十分简单, 通过`block_id`和`offset`屏蔽`DiskInode`的三级指针, 即屏蔽了硬件细节. 我们可以把`EfsInode`和`DiskInode`理解成easy-fs这个文件系统管理它的块设备的策略, 不同的文件系统根据实际应用场景的不同, 在管理块设备的策略选择上肯定是有所侧重的, 比如有一些侧重稳定性, 有一些更加侧重性能等等. EfsInode驻留在内存中, 屏蔽了块设备的硬件细节, 以及提供DiskInode的解析结果的缓存和一些运行时的状态管理(目前还没实现), DiskInode则驻留在磁盘中, 提供对文件元信息的持久化抽象描述.

之后, 我们就可以通过这个结构体去实现这个文件系统对块设备的管理策略:

```rust
impl EfsInode {
    /// We should not acquire efs lock here.
    pub fn new(
        block_id: u32,
        block_offset: usize,
        fs: Arc<Mutex<EasyFileSystem>>,
        block_device: Arc<dyn BlockDevice>,
        inode_id: u32
    ) -> Self {
    }

    // lambda函数会捕获到&DiskInode
    fn read_disk_inode<V>(&self, f: impl FnOnce(&DiskInode) -> V) -> V {
        get_block_cache(self.block_id, Arc::clone(&self.block_device))
        //  ↑ 拿到磁盘块缓存（512 字节）
            .lock()
        //  ↑ 互斥访问
            .read(self.block_offset, f)
        //  ↑ 在 block_offset 偏移处，把字节强行解读为 &DiskInode
        //    然后交给闭包 f 使用，返回 f 的返回值
    }

    fn modify_disk_inode<V>(&self, f: impl FnOnce(&mut DiskInode) -> V) -> V {
        get_block_cache(self.block_id, Arc::clone(&self.block_device))
            .lock()
            .modify(self.block_offset, f)
    }
    
    fn find_inode_id(&self, name: &str, disk_inode: &DiskInode) -> Option<u32> {
    }

    // self是一个目录, 查找目录下的文件
    pub fn find(&self, name: &str) -> Option<Arc<EfsInode>> {
    }

    // 判断是否是目录
    pub fn is_dir(&self) -> bool {
        self.read_disk_inode(|disk_inode| disk_inode.is_dir())
    }

    fn increase_size(
        &self,
        new_size: u32,
        disk_inode: &mut DiskInode,
        fs: &mut MutexGuard<EasyFileSystem>,
    ) {
    }
    /// 在本目录下创建一个新节点 (文件、目录)
    pub fn create(&self, name: &str, type_: DiskInodeType) -> Option<Arc<EfsInode>> {
    }

    pub fn ls(&self) -> Vec<String> {
    }

    pub fn read_at(&self, offset: usize, buf: &mut [u8]) -> usize {
    }   

    pub fn write_at(&self, offset: usize, buf: &[u8]) -> usize {
    }

    pub fn clear(&self) {
    }
}
```

到这里, 底层的工作已经基本完成了, 剩下的就是对接到上层的VFS中间层抽象了. 中间层抽象就意味着它定义的是一套接口, 而不是具体的实现. 因此, 我们用trait来描述VFS所需要的接口:

```rust
pub enum ErrorNo {
    ENOENT,   // No such file or directory
    EISDIR,   // Is a directory
    ENOTDIR,   // Not a directory
    ENOSYS,   // Not supported
    EBUSY,    // Device or resource busy
    EINVAL,   // Invalid argument
    EEXIST,   // File exists already
}

pub struct Metadata {
    is_dir: bool,
    size: usize,
    /* 后续有更多字段 */
}

pub trait VfsNode: Send + Sync {
    fn inode_id(&self) -> u32;

    fn metadata(&slf) -> Result<Metadata, ErrorNo>;

    fn file_size(&self) -> Result<usize, ErrorNo> {
        self.metadata().map(|m| m.size())
    }

    fn is_dir(&self) -> bool {
        self.metadata().map(|m| m.is_dir()).unwrap_or(false)
    }
    fn is_file(&self) -> bool {
        self.metadata().map(|m| m.is_file()).unwrap_or(false)
    }

    fn read_at(&self, offset: usize, buf: &mut [u8]) -> Result<usize, ErrorNo>;
    
    fn write_at(&self, offset: usize, buf: &[u8]) -> Result<usize, ErrorNo>;

    /* 默认是普通文件映射 */
    fn mmap_type(&self) -> MMapType {MMapType::FileBacked}
    
    // default implementation
    fn find(&self, name: &str) -> Result<Arc<OSInode>, ErrorNo> {Err(ErrorNo::ENOSYS)}
    
    fn create(&self, name: &str, node_type: NodeType) -> Result<Arc<OSInode>, ErrorNo> {Err(ErrorNo::ENOSYS)}
    
    fn unlink(&self, name: &str) -> Result<(), ErrorNo> {Err(ErrorNo::ENOSYS)}
    
    fn readdir(&self, offset: usize) -> Result<Option<DirEntry>, ErrorNo> {Err(ErrorNo::ENOSYS)}
    
    fn truncate(&self) -> Result<(), ErrorNo> {Err(ErrorNo::ENOSYS)}

    fn get_mmap_info(&self, offset: usize, length: usize) -> Result<MMapInfo, ErrorNo>;
}
```

这个trait其实混杂了元数据和文件操作接口, 是一个"集大成"的trait, 和linux中分成几个结构体去实现不同, 但是我们不是用于生产中的操作系统, 因此只需要明白这些东西的核心思想就好, 不用实现的尽善尽美. 现在, 底层不同的文件系统只需要去实现这个trait, 就可以对接到上层了, 上层也只需要去调用trait里面提供的接口, 而不需要了解底层的细节. easy-fs去实现这个trait:

```rust
pub struct EfsInodeWrapper {
    inner: Arc<EfsInode>,
}

impl EfsInodeWrapper {
    pub fn new(inner: Arc<EfsInode>) -> Self {
        Self { inner }
    }

    /// 获取内部 easy-fs inode 引用，用于需要直接操作 easy-fs 的场景（如根目录获取）
    pub fn inner(&self) -> &Arc<EfsInode> {
        &self.inner
    }
}

impl VfsNode for EfsInodeWrapper {
....
}
```

其实就是去调用EfsInode里impl的那些方法, 技术上并不难, 但是这个思想是十分伟大的.

VfsNode是一个trait, 不存在内存实体, 描述的是所有要接入VFS中间层抽象的具体模块所需要实现的接口. 为了统一管理千变万化的底层模块, 我们还需要一个统一的实体来包装背后的具体模块并暴露给上层, 这样子, 上层就仅仅需要和这个统一的实体做交互, 而不用关心实体背后的具体模块是什么. 这种做法就像Linux系统中的Inode和Inode_info, 这两者的背后可能是ext4文件系统, 或者是tmpfs, sysfs, devfs等驻留在内存中的文件系统, 但是上层调用者大多数情况下并不用关心这个细节. 我们引入OSInode来统一包装背后的模块:

```rust
/* VFS Inode实体 */
/* 这个是暴露给用户态用的, read和write方法实际上调用的是entity在VfsNode trait中实现的方法 */
pub struct OSInode {
    pub inode_id: u32, 
    pub dev_id: u32,
    pub metadata: Metadata,
    pub entity: Arc<dyn VfsNode>
}

// 就是去无脑调用entity的方法, 相当于一个wrapper
impl VfsNode for OSInode {
    fn xxx(&self, xxx, ...) -> xxx {
		    self.entity.xxx()
    }
}

lazy_static! {
    pub static ref ROOT_INODE: Arc<OSInode> = {
        let efs = EasyFileSystem::open(BLOCK_DEVICE.clone());
        let root_efs = EasyFileSystem::root_inode(&efs);
        let wrapper = EfsInodeWrapper::new(Arc::new(root_efs));
        let ino = wrapper.inode_id();          // 根目录 inode 号为 0
        let meta = wrapper.metadata().unwrap(); // is_dir=true, size 从磁盘读
        Arc::new(OSInode::new(ino, 0, meta, Arc::new(wrapper)))
    };
}
```

这样子, OSInode就相当于一个stub, 负责承接用户传过来的请求, 把请求透传到其背后的实体.
