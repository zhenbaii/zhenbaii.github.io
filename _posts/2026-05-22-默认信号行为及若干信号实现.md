> 实现默认信号行为, 以及实现若干信号
#### 默认动作

在POSIX标准下, 不同信号的默认行为是不同的, 这些默认行为通常都可以归类为以下enum类型所表达的行为:

```rust
// os/src/task/action.rs
pub enum DefaultAction {
    Ignore,                 // Clear the pending bit of the signal without any action
    Stop,                   // Pending the current process
    Terminate,              // Terminate the current process
    Continue,               // Continue the current process if it is pending
}
```

因此,实现默认信号动作的第一步,就是实现一个工具函数,把对应信号路由到对应的默认动作, 见`os/src/task/signal.rs`的`pub fn default_action(&self) -> DefaultAction`.实现了这个函数后,对于某个具体信号,我们就知道该为它执行什么默认行为. 因此, 我们实现执行默认行为函数如下:

```rust
fn do_default_action(signal: SignalFlags) {
    let task = current_task().unwrap();
    let mut task_inner = task.inner_exclusive_access();
    let default_action = signal.default_action();
    match default_action {
        DefaultAction::Ignore => {
            // ignore it, no extra action
            task_inner.signals ^= signal;
        }
        DefaultAction::Stop => {
            // stop the process
            task_inner.frozen = true;
            task_inner.signals ^= signal;
        }
        DefaultAction::Terminate => {
            // terminate the process
            task_inner.killed = true;
        }
        DefaultAction::Continue => {
            // continue the process if it is pending
            if task_inner.frozen {
                task_inner.frozen = false;
            }
            task_inner.signals ^= signal;
        }
    }
    /* 仅设置tcb,实际执行动作不在这里 */
}
```

需要注意的是,这个函数实际上并不会真的执行那些默认动作, 而仅仅只是设置标志位, 信号动作的真正执行都是发生在进程从用户态trap到内核态时执行的, 即`trap_handler`中. 也就是说,用户态进程在发送一个信号给另一个进程时,那一刻所做的事,仅仅就是给那个进程的TCB的signal置位,真正的执行时机是对应进程的trap上下文中:

```rust
pub fn trap_handler() -> ! {
		.......
		// 某个进程在trap时会顺便处理信号，也就是说某个进程向另外一个进程发送信号，并不会马上触发其信号处理
		// 而是等待该进程trap(主动发起或者进程调度)时才会处理一次
		// 这个函数执行完后，进程要不frozen,要不其trap_ctx被修改
		handle_signals();
		
		
		// check error signals (if error then exit)
		if let Some((errno, msg)) = check_signals_error_of_current() {
		    println!("[kernel] {}", msg);
		    exit_current_and_run_next(errno);
		}
		trap_return();
}
```

handle_signals函数内部是一个循环,循环内部会不断检查当前进程中被置位的信号位, 过滤出可以被安全执行的信号(当前进程没有屏蔽这个信号, 并且这个信号没有被当前正在执行的信号屏蔽)后, 执行这个信号的回调函数. 

```rust
// os/src/task/mod.rs
pub fn handle_signals() {
    loop {
        check_pending_signals();    /* 如果确实需要处理信号，经过这个调用后，trap_ctx就会被修改好 */
        let (frozen, killed) = {
            let task = current_task().unwrap();
            let task_inner = task.inner_exclusive_access();
            (task_inner.frozen, task_inner.killed)
        };
        // println!("Process frozen state: {}", frozen);
        /* 如果当前进程被杀死或者没有被冻结，就不用反复挂起自己了 */
        if !frozen || killed {
            break;
        }
        // println!("Process frozen state: {}", frozen);
        suspend_current_and_run_next();
    }
}
```

对于那些在用户态注册了回调函数的信号, 其原本的TCB会被备份, 然后TCB的sepc寄存器的地址会被写成用户态回调函数的地址,这样子,在后续回到用户态的时候,就会自然而然地执行用户态的回调函数. 之后,在回调函数中再次执行一个特殊的系统调用陷回内核,做正常的现场恢复,整个信号执行过程也就完成了:

```rust
/// 执行用户态信号回调函数
/// 总的来说就是修改trap_ctx，主要修改内核态返回用户态时的返回地址(sepc)，以及回调函数所需的参数
fn call_user_signal_handler(sig: usize, signal: SignalFlags) {
    let task = current_task().unwrap();
    let mut task_inner = task.inner_exclusive_access();

    let handler = task_inner.signal_actions.table[sig].handler;   /* 用户态信号处理函数的地址 */
    if handler != SIG_DFL {
        // user handler
        // 如果是SIG_IGN
        if handler == SIG_IGN {
            task_inner.signals ^= signal;
            return;
        }
        // handle flag
        task_inner.handling_sig = sig as isize;    /* 标记当前信号为当前进程正在处理的信号 */
        task_inner.signals ^= signal;

        // backup trapframe
        let trap_ctx = task_inner.get_trap_cx();
        task_inner.trap_ctx_backup = Some(*trap_ctx);    /* 备份当前TrapContext，用于恢复现场 */

        // modify trapframe
        trap_ctx.sepc = handler;    /* 设置当前的中断上下文的用户态返回地址为handler,这样子返回用户态时就会执行用户态的handler了 */

        // put args (a0)
        trap_ctx.x[10] = sig;      /* 用户态信号回调的参数为信号编号 */
    /* 进入这个分支，意味着某个信号的pending位被置1(kill系统调用)，但是该进程没有为这个信号注册回调(没有调用sigaction) */
    } else {
        drop(task_inner);    /* do_default_action中会获取锁 */
        // default action
        do_default_action(signal);
        // println!("[K] task/call_user_signal_handler: default action: ignore it or kill process");
    }
}
```

可以看到, `do_default_action`在else分支被执行.

#### **实现 SIGCHLD & SIGPIPE**

SIGCHLD信号产生的条件主要有以下几个：

- 子进程终止时；
- 子进程接收到SIGSTOP信号停止时；
- 子进程处在停止态，接受到SIGCONT后唤醒时；

即子进程的执行状态发生变化时,就会发送这个信号去通知父进程. 这个信号的默认行为一般是忽略(默认忽略),也就是操作系统并不会主动介入子进程状态变化后的动作, 这个时候就要依赖父进程调用wait系统调用等待子进程安全退出后, 回收子进程占用的一点系统记录(主要是TCB), 防止成为僵尸进程. 

然而, wait系统调用是阻塞的, 也就是说父进程会阻塞在这个调用中,直到子进程退出, 这会极大限制父进程的手脚. 通过在用户态注册SIGCHLD回调函数, 在回调函数中做子进程的回收, 我们就可以达到异步回收子进程资源的效果.

实现这个信号并不难, 就是在子进程死掉(`exit_current_and_run_next`)的上下文中, 插入对父进程这个信号的判断:

```rust
pub fn exit_current_and_run_next(exit_code: i32) {
    ........
    // 发送SIGCHLD信号给父进程
    if let Some(parent_weak) = &task.inner_exclusive_access().parent {
        if let Some(parent) = parent_weak.upgrade() {
            let mut parent_inner = parent.inner_exclusive_access();
            // 添加针对SIGCHLD的SIG_IGN的判断
            if parent_inner.signal_actions.table[SignalFlags::SIGCHLD.sig_num()].handler == SIG_IGN {
                // 父进程显式忽略 SIGCHLD，内核自动回收子进程，不留僵尸
                parent_inner
                    .children
                    .retain(|c| c.getpid() != task.getpid());
            } else {
                parent_inner.signals |= SignalFlags::SIGCHLD;
            }
        }
    }
    // **** release current PCB
    // drop task manually to maintain rc correctly
    drop(task);
}
```

可以发现, 代码中还添加了SIG_IGN的判断, 如果父进程把其回调设置为这个, 即所谓的显式忽略, 就会由操作系统代劳回收子进程的TCB, 从而不会产生僵尸进程. 

在默认情况下(即什么都不做), 内核并不知道父进程对于子进程的态度是怎么样的, 因此, 为了最大限度地保证系统稳定性, 内核会把子进程的TCB的裁判权交给程序员,也就是说,这个时候依赖我们主动调用wait()获取和感知子进程状态,并做资源的回收,操作系统不会盲目介入帮我们回收.

在设置SIG_IGN后,相当于告诉内核,父进程对于其子进程的死活一概不管,操作系统你帮我收尸就好, 因此, 在设置了这个后,我们就会在内核上下文中就把TCB提前回收, 防止产生僵尸进程.

SIGPIPE信号是在管道读端关闭后, 内核发送信号给写端, 通知写端不用再向管道中写东西了.  

首先,要添加一个Weak引用的读端, 用来让写端感知读端的变化:

```rust
pub struct PipeRingBuffer {
    arr: [u8; RING_BUFFER_SIZE],
    head: usize,
    tail: usize,
    status: RingBufferStatus,
    write_end: Option<Weak<Pipe>>,
    read_end: Option<Weak<Pipe>>   // 写端用这个感知读端的存在与否
}
```

之后, 在write函数时, 写端在尝试向管道写入数据时, 如果发现写不进去, 就可以判断,是因为写完了,还是因为读端关闭.如果读端关闭,就由内核向这个进程发送SIGPIPE信号:

```rust
if loop_write == 0 {
    if ring_buffer.all_read_ends_closed() {
        /* 检测到读端关闭，此时内核会发送给进程SIGPIPE信号，提示不用再写了 */
        current_add_signal(SignalFlags::SIGPIPE);
        return already_write;
    }
    drop(ring_buffer);
    suspend_current_and_run_next();
    continue;
}
```
